# AI 周报生成技能

用于每周一自动生成一篇结构化的 AI 行业周报，保存到 `posts/` 目录下。

---

## 用途

- 追踪每周 AI 行业最重要的新闻和趋势
- 按照统一的主题分类整理信息
- 输出模板中规定的 Markdown 文档
- 保持周报风格、结构和命名的一致性

---

## 使用时机

当用户要求"生成本周 AI 新闻"、"做一期 AI 周报"、"更新 AI 周报"或类似指令时调用本技能。

---

## 工作流程

### 1. 确定发布日期

- 周报固定每周一发布
- 标题格式：`本周 AI 新闻 YYYYMMDD`
- 文件命名：`posts/YYYY/YYYY-MM-DD-ai-weekly.md`
- 示例：2026 年 7 月 13 日周一，标题为 `本周 AI 新闻 20260713`，文件为 `posts/2026/2026-07-13-ai-weekly.md`

### 2. 信息采集

使用 WebSearch 等工具搜索当周 AI 相关新闻。建议按九大主题分别搜索，每个主题至少尝试 2-3 个不同角度（例如"主题名 + 本周日期"、"主题名 + 头部公司名"、"主题名 + 融资/发布/监管"），以确保每个主题下有足够多的独立事件可选。重点覆盖：

* **海外头部 AI 公司**：OpenAI、Anthropic、Google、Microsoft、xAI、Meta、NVIDIA、Mistral AI、Apple
* **国内头部 AI 公司**：阿里、腾讯、百度、字节、华为、快手、智谱 AI、DeepSeek、月之暗面、MiniMax、零一万物、百川智能
* **权威科技媒体**：TechCrunch、The Verge、Reuters、WSJ、Bloomberg、财新、晚点 LatePost、36氪、虎嗅
* **AI 专业媒体**：The Information、MIT Technology Review、VentureBeat、AI News、机器之心、量子位、新智元、腾讯研究院
* **AI 专业社区**：Hacker News、Reddit r/MachineLearning、Reddit r/LocalLLM、Reddit r/ArtificialIntelligence、Hugging Face、GitHub Trending

**时间窗口规则（强制）**：

- 周报发布日为周一，内容覆盖**上周一至周日**之间首次公开报道或取得**实质性进展**的 AI 事件。
- 使用 WebSearch 时，优先按日期过滤；每条入选新闻必须在正文中明确标注公开日期。
- 上周一至周日范围外发生的事件，若其影响在本周持续发酵或本周有重要后续进展，可纳入本周范围。
- 非本周事件仅在以下两种情况下可提及，且不得超过一句话：
  1. 作为本周事件的必要背景；
  2. 本周有了明确的后续进展（如正式发布、交易完成、政策生效）。

### 3. 内容填充

按九大主题填充内容。每个主题下用 `### 1. / 2. / 3.` 编号。每条新闻的推荐结构为：

1. **事件段落**：用 1 段话说明发生了什么，包含必要的时间、参与方与核心动作。
2. **核心数据 bullet**：以 `- **核心数据**：...` 的形式列出具体数字、规模、关键事实。
3. **关键信号 bullet**：以 `- **关键信号**：...` 的形式说明这件事为什么重要、释放了怎样的趋势信号。

**一事一号规则（强制）**：

- 每个 `### 1. / 2. / 3.` 编号下**只能写一个独立事件**。
- 如果同一主题下有多个相关但独立的事件（例如同一家公司发布两款产品、两个不同公司发布同类产品），必须分别编号，不得用"同期/同时/此外"等词把多个事件拼接成一段。
- 判断标准：如果一个段落里出现了两个以上的"日期 + 主体 + 动作"组合，就必须拆分为多个编号。

**主题唯一归属规则（强制）**：

- 每条新闻只能在一个主题下作为 `### 1. / 2. / 3.` 详细展开。
- 归属以新闻的**核心动作**为准：
  - 收购/发布编程工具 → **编程与开发工具**
  - 模型被政府下线/监管事件 → **监管与合规动态**
  - 模型能力发布/评测 → **大模型前沿动态**
  - 融资/并购交易 → **投资融资与商业**，但若交易标的是某一主题下的具体产品/公司，则优先归属该产品/公司所属主题，投资金额在段落内带过即可。
- 当一条新闻可能跨多个主题时，选择**最稀缺、最本周、最能说明趋势**的主题。
- 其他主题如需引用该事件，只允许用一句话交叉索引，例如："详见'三、编程与开发工具'。"

### 4. 精简与规范

- **时间过滤**：逐条检查已收集的新闻，凡公开日期不在上周一至周日范围内、且不属于本周实质性进展的事件，必须剔除，不得作为编号条目保留。
- **动态剪枝与重排**：
  1. 若某个主题在当周完全没有可靠信源，请**连同注释整段删除，不要硬凑**。
  2. 若某个主题只有 1 条新闻，可以保留，并允许在主题引言中用一句话带过相关背景。
  3. 若某个主题有 2 条及以上独立新闻，必须分别编号为 `### 1. / 2. / 3.`，禁止把多个事件融合成一条。
  4. 删除主题后，必须对剩余的主题按“一”至“九”的顺序重新进行标准汉字编号，确保结构连贯。
  5. 目标：每个保留的主题尽量有 2-3 条独立新闻；若实在不足，宁可删除主题，也不要把多个事件融合成一条。
- **去重与唯一归属**：确保同一条新闻只在一个主题下详细展开；若发现重复，保留最符合核心动作的主题，其余位置用一句话交叉索引或删除。
- **看点联动**：顶部“本周看点”的内容必须是本周最重要的 3-5 条核心观察，与正文最终保留的核心事件严格对应，但不要求与九大主题一一对应。

### 5. 参考来源

所有参考来源必须提供可点击的链接。格式：

```markdown
- [来源名称](URL)
```

## 命名与格式规范

### 文件名

```
posts/YYYY/YYYY-MM-DD-ai-weekly.md
```

### 内容与结构规范

生成新周报时，**必须严格复制并基于 `template.md` 模板文件进行填充**，并遵循以下规范：

1. **Front Matter 填充**：完整继承模板顶部的 YAML 配置。
   - `tags` 必须使用标准 YAML 数组格式，例如 `[ai-weekly, <其他推断标签>]`，总量不超过 8 个。
   - `excerpt` 必须使用一句话提炼核心趋势。

2. **本周看点**：采用 3-5 条编号的核心观察，每条包含一个吸引人的小标题和一句话的事件+影响说明。不要与九大主题机械一一对应，而应突出本周最重大的变化。

3. **正文骨架**：严禁修改或擅自增减模板中固定的标准 Markdown 标题和分割线。

4. **新闻条目结构**：每个 `### 1. / 2. / 3.` 下先写事件段落，再跟 `- **核心数据**：...` 和 `- **关键信号**：...` 两个 bullet。

#### tags 生成规则

生成 `tags` 时，按以下步骤处理：

1. **推断标签**：根据工具功能、技术栈和主题，推断出一组候选标签。优先使用小写英文标签（如 `anthropic`、`openai`、`agent`），必要时可混合中文标签。
2. **复用已有标签**：读取项目根目录的 `tags.txt`，在候选标签中优先使用已存在的标签。注意 `tags.txt` 中的标签格式为每行一个，读取时需按行处理。
3. **追加新标签**：如果某个候选标签在 `tags.txt` 中不存在，则将其追加到 `tags.txt` 文件末尾，**每个标签独占一行**。
4. **最终 frontmatter**：`tags` 字段必须包含 `ai-weekly`，并按字母或语义顺序排列，例如：

   ```yaml
   tags: [ai-weekly, openai, anthropic, agent]
   ```

> 注意：不要重复添加 `tags.txt` 中已有的标签；追加新标签时不要在行尾添加多余标点或空行。

## 注意事项

1. 周报固定周一发布，标题和文件名必须对应当周周一日期
2. 内容覆盖上周一至周日的 AI 新闻
3. 每个主题下没有内容时可省略该主题，不要硬凑；保留的主题尽量有 2-3 条独立新闻
4. 数字、金额、百分比统一使用阿拉伯数字
5. 避免使用英文术语替代中文表达（如 crossover 应改为"交叉点"或"拐点"）
6. 参考来源必须提供可点击的真实链接，禁止捏造 URL。
7. 保持客观中立，不做过度推断
8. 排版遵循中英文、数字间空格规范
